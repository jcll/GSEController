## Critical ‚Äî Fix Before Next Release

### BUG-001: `stop()` resets KeyHelper state, causing a blocking `O_WRONLY` open and silent key drops on every Stop/Start cycle

**Files:** `ControllerManager.swift:265`, `KeySimulator.swift:105‚Äì113`, `338‚Äì355`

Every click of Stop calls `KeySimulator.stopHelper()`, which closes the write fd and resets `_setupStarted = false`. On the next Start, `ensureHelper()` dispatches work to a background GCD thread that calls `openFIFO()` with `O_WRONLY`. POSIX semantics: `open(O_WRONLY)` on a named pipe **blocks until a reader opens the other end**. The launchd agent restarts after the write-end close, but there is a multi-second window where the background thread is stuck at `open()`. During that window, `_fd` is still `-1`, so every button press silently drops its key event ‚Äî the app shows "Active" but sends nothing.

**Fix:** Do not call `stopHelper()` from `stop()`. The launchd agent stays running across Stop/Start anyway ‚Äî keep the FIFO write fd open. Only call `stopHelper()` on app quit (e.g., `applicationWillTerminate`). This makes `ensureHelper()` a no-op on subsequent Start presses.

### BUG-002: `axMonitorTask` sleeps 3 seconds before its first Accessibility check

**File:** `FireEngine.swift:51‚Äì65`

The `axMonitorTask` loop starts with `try? await Task.sleep(for: .seconds(3))`. The first AX check is therefore deferred 3 seconds. If Accessibility permission is revoked at the moment firing begins (e.g., after a macOS security update resets TCC), the engine continues sending key events for up to 3 seconds while displaying "FIRING" ‚Äî macOS silently discards them.

**Fix:** Move the sleep to the end of the loop so the first check runs immediately:

```swift
axMonitorTask = Task { [weak self] in
    while !Task.isCancelled {
        guard let self, !Task.isCancelled else { break }
        let ok = KeySimulator.isAccessibilityEnabled
        self._axEnabled.withLock { $0 = ok }
        if !ok {
            Self.logger.warning("Accessibility permission revoked, stopping")
            self.stopAll()
            self.onAccessibilityRevoked?()
            break
        }
        try? await Task.sleep(for: .seconds(3))
    }
}
```

## Moderate ‚Äî Should Fix

### ARCH-004: `DualSenseBatteryMonitor` shares a single IOKit report buffer across all attached devices ‚Äî data race with two controllers

**File:** `DualSenseBatteryMonitor.swift:41`, `89‚Äì96`

`private let buf = UnsafeMutablePointer<UInt8>.allocate(capacity: 256)` is passed as the report buffer to `IOHIDDeviceRegisterInputReportCallback` for **every** attached DualSense device. If two DualSense controllers (e.g., DualSense + DualSense Edge, both in `productIDs`) fire HID input reports concurrently, both C callbacks write into the same 256-byte buffer simultaneously ‚Äî completely unsynchronized. The class is marked `@unchecked Sendable` which suppresses compiler warnings about this.

**Fix (minimal):** Allocate a separate buffer per device in `attach(_ device:)` and free it in the device-removal callback. Alternatively, read from the `data` pointer that IOKit passes directly to the callback handler rather than from `buf` ‚Äî `buf` is only needed as the registration-time buffer argument.

### BUG-003: `Task.detached` in `checkAccessibility()` captures `self` strongly and blocks a cooperative thread pool thread

**File:** `ControllerManager.swift:339‚Äì343`

```swift
Task.detached(priority: .userInitiated) {
    let helperAx = KeySimulator.isHelperAccessibilityEnabled
    await MainActor.run { self.hasHelperAccessibility = helperAx }
}
```

`isHelperAccessibilityEnabled` runs a subprocess via `Process` and calls `proc.waitUntilExit()` ‚Äî a blocking synchronous call inside a Swift concurrency task. This holds a cooperative thread pool thread for the subprocess duration. The method is triggered by every `NSApplication.didBecomeActiveNotification` (line 137), so rapid Cmd-Tab cycling piles up multiple blocked threads. The strong `self` capture also retains `ControllerManager` for the life of each pending task.

**Fix:**
```swift
Task.detached(priority: .userInitiated) { [weak self] in
    let helperAx = KeySimulator.isHelperAccessibilityEnabled
    await MainActor.run { self?.hasHelperAccessibility = helperAx }
}
```
Consider also debouncing the activation notification or guarding with an in-flight flag to prevent concurrent runs.

### ARCH-002: `KeySimulator` is an untestable static singleton that silently swallows all setup failures

**File:** `KeySimulator.swift`

`KeySimulator` is an `enum` with all-static state, making it impossible to mock or inject. More critically, `ensureHelper()` dispatches fire-and-forget to a global queue and logs failures via `logger.error` ‚Äî callers have no way to know whether the helper actually came up. `ControllerManager.start()` calls `ensureHelper()` and immediately sets `isRunning = true` without waiting for or checking the result. If Xcode Command Line Tools are absent, the C compile fails silently, and the app reports "Active" while sending zero keystrokes.

**Recommended change:** Extract a `KeySimulatorService` protocol with `isReady: Bool`, `isAccessibilityEnabled: Bool`, and `ensureHelper() async throws`. Give `ControllerManager` a concrete instance. Gate `isRunning = true` in `start()` on `await ensureHelper()` succeeding, and surface failure in `statusMessage`.

### ARCH-001: `ControllerManager` caches a stale `activeGroup` snapshot that can diverge from `ProfileStore`

**File:** `ControllerManager.swift:30`, `121`

`ControllerManager` stores `activeGroup: ProfileGroup?` ‚Äî a snapshot of the group passed to `start(group:)`. On controller reconnect, `attachHandlers(for: group)` rebuilds from this cached copy (line 121). If the user edits and saves the group in `ProfileStore` while running, `activeGroup` diverges from the stored version. The Save button being disabled while running (ContentView line 78) papers over this mismatch rather than fixing the root cause.

**Fix:** Remove `activeGroup` from `ControllerManager`. At `start(group:)` time, derive the handler set (a `[ControllerButton: MacroBinding]` map) and store only that. Pass the group name as a label string for `statusMessage`. `ProfileStore` becomes the single source of truth.

### ARCH-003: Both `@StateObject` singletons live in `ContentView`, not the scene ‚Äî wrong object lifetime

**File:** `ContentView.swift:4‚Äì5`, `GSEControllerApp.swift`

`ContentView` instantiates `@StateObject private var store = ProfileStore()` and `@StateObject private var controller = ControllerManager()`. With the current single `Window` scene this is harmless, but it means the singletons' lifetimes are tied to the view hierarchy rather than the app scene. If the scene ever becomes a `WindowGroup` or a second window is opened, each `ContentView` gets its own `ControllerManager` and `ProfileStore` ‚Äî duplicating `GCController` observers, `DualSenseBatteryMonitor`, and launchd agent checks.

**Fix:** Move both `@StateObject`s to `GSEControllerApp` and inject them via `.environmentObject(_:)`. `ContentView` becomes a pure consumer. Minimal change; correct semantics.
